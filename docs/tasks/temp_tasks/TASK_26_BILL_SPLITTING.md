## Точная реализация (Technical Design)

### 1. БД миграция и сущности

#### V13__create_bill_splits_table.sql
```sql
CREATE TABLE IF NOT EXISTS bill_splits (
    id UUID PRIMARY KEY,
    order_id UUID NOT NULL,
    item_id UUID NOT NULL, -- MenuItem UUID
    quantity INT NOT NULL,
    amount DECIMAL(10, 2) NOT NULL,
    status VARCHAR(50) NOT NULL DEFAULT 'PENDING', -- PENDING, PAID
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX IF NOT EXISTS idx_bill_splits_order ON bill_splits(order_id);
```

### 2. Логика разделения счета (PaymentService.java)
- Когда гость разделяет чек, он отправляет массив позиций (itemId + quantity).
- Бэкенд проверяет, не были ли эти позиции уже оплачены другими частями разделения (считает сумму `quantity` в `bill_splits` со статусом `PAID` для данного `itemId`).
- Создает запись `BillSplitEntity` со статусом `PENDING`.
- При подтверждении оплаты этой части (официантом или картой):
  - Переводит статус `BillSplitEntity` в `PAID`.
  - Проверяет: если сумма всех оплаченных частей (`bill_splits`) равна полной сумме заказа (или все позиции заказа полностью покрыты оплаченными `bill_splits` по количеству), то автоматически переводит основной заказ (`OrderEntity`) в статус `PAID` и освобождает столик.

### 3. Frontend Split UI (`app/(guest)/receipt/page.tsx`)
- По умолчанию отображается весь чек столика.
- При включении тумблера «Разделить счет»:
  - Возле каждой позиции чека появляется счетчик (например, если в заказе 4 пиццы Маргариты, гость может выбрать стрелочками «1» или «2» штуки, которые он берет на себя).
  - Позиции, которые уже полностью оплачены другими гостями за столом (информация подтягивается с бэкенда из списка оплаченных `bill_splits`), блокируются для выбора и помечаются серым цветом «Оплачено кем-то».
  - Внизу чека динамически рассчитывается сумма: `Ваша доля: 18.50 BYN` (с учетом выбранных модификаторов и размеров).
  - Кнопка «Оплатить мою долю» вызывает API `/split` и перенаправляет на оплату (или вызывает официанта именно для оплаты этой части).

## Риски и подводные камни (Edge Cases)

- **Округление копеек при делении:** Если гости делят одно блюдо пополам (например, 1 пицца на двоих). В API разрешить делить количество как дробное число (double/BigDecimal) или просто разрешить делить общую сумму. Лучше делить поштучно (целыми числами позиций), это намного проще и исключает ошибки округления. Если гости хотят поделить одно блюдо пополам, они могут попросить официанта разделить чек вручную на POS-терминале. В PWA делаем поштучное разделение.
- **Оплата одной и той же позиции дважды:** Строго валидировать на бэкенде остаток неоплаченного количества для каждого блюда в заказе перед созданием `BillSplit`.

## Ручная проверка (Гайд для пользователя/разработчика)

1. Сделать заказ гостем: 2 пиццы Маргариты (по 18.50 BYN) и 2 Кока-колы (по 3.00 BYN). Всего: 43.00 BYN.
2. Открыть страницу чека гостя `/receipt`.
3. Включить тумблер «Разделить счет».
4. Выбрать: 1 пиццу Маргариту и 1 Колу. Внизу должна отобразиться сумма «Ваша доля: 21.50 BYN».
5. Нажать «Оплатить мою долю» (выбрать оплату наличными через официанта).
6. В панели официанта `/dashboard` подтвердить оплату этой части чека.
7. Открыть чек гостя повторно — выбранные ранее 1 пицца и 1 кола должны быть заблокированы с пометкой «Оплачено». Доступны для выбора оставшиеся 1 пицца и 1 кола.
8. Выбрать оставшиеся позиции и оплатить их.
9. Убедиться, что после оплаты второй части весь заказ перешел в статус `PAID` и столик освободился.
